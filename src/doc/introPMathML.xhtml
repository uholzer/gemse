Introduction to Presentation MathML

Mathematicians usually have good knowledge of LaTeX. Because this is probably not true for MathML and because I want mathematicians to be able to use Gemse, I give here a very short introduction to Presentation MathML. However, this introduction scratches only the surface of Presentation MathML, so you want to look into the MathML specification afterwards. Additionally, there is the :help element command to open the specification for the element under the cursor. To look up the specification for a specific element, say mrow, use :help element mrow.

MathML

MathML is a markup language based on XML stanardized by the W3C. It consists of two complementary parts, Presentation Markup and Content Markup. If we want to refer to the Presentation Markup of MathML we often just call it Presentation MathML. Similarly, we often call the Content Markup simply Content MathML.

The Presentation Markup captures the structure of a written formula. It describes which characters are identifiers, operators, and numbers and defines how they visually relate to each other. The Presentation Markup might for instance say that there is a fraction and above the bar of the fraction is an identifier "a" and below is the number 2. The exact rendering may differ from device to device. The formulas might be read aloud by a text to speech system or a part of them might be copy-pasted into another application. A formula can be integrated in an HTML document on a website. Its appearance (font, colour, background, borders) can be influenced with CSS the same way an HTML document is styled. Finally, the formula can be made interactive using JavaScript.

The Content Markup captures the actual meaning of a formula as opposed to its visual appearance. For instance, fx might denote the function with index x of a set of functions in some documents while it denotes the derivative in x in other texts. The Content Markup explicitely and precisely tells which meaning is intended, making it clearly understandable to mathematical software. But the Content Markup does not tell how to write the formula down, giving no instructions on how to present the formula to a human. Content Markup cannot easily be converted to Presentation Markup because how a formula is written very much depends on the context and conventions established in a document or field of research. However, it is possible to formulate rules for such a conversion, albeit not with MathML itself.

For comparison, LaTeX is a complete typesetting system. The LaTeX code we write is an actual program that outputs a typeset document. The output is usually a PDF which precisely describes the shape and location of each character to reproduce it exactly the same in print as on screen. MathML is much more verbose than LaTeX as it has to capture more information which LaTeX figures out on its own. On the other hand, MathML can be used in many more diverse ways than just displaying and printing. The cost of this is that MathML is more of a nuisance to author. Writing MathML by hand with a plain text editor is not advisable.

Unicode

In order to store and process text on a computer, one needs a convention on how to represent text inside the memory of the computer. The most popular convention is the Unicode standard. It assigns to each character a unique number, called the Unicode code point. Unicode includes Latin characters but also characters from almost all scripts used around the world. This includes the Greek alphabet and a plethora of mathematical symbols. For example, the rightward arrow → is assigned the number 8594 (or 0x2192 in hexadecimal notation). Instead of giving names to characters (e.g. \rightarrow as you would do in LaTeX), MathML directly uses the characters from Unicode. As a consequence, you will often come into contact with Unicode while using Gemse. For example, if there is no easier way, Gemse will always allow you to insert any character by entering its Unicode code point, which you can lookup in the Character Code Charts of the Unicode standard. Additionally, Gemse obtains information about characters (e.g. if its an operator or identifier) from the Unicode standard.

Those parts that begin with a < and end with the next > are called tags. There are two sorts of tags: Start tags and end tags. Those that begin with </ are the end tags, the others are start tags. For example, <mrow> is a start tag and </mrow> is an end tag. (There is also an empty-element tag, which is a start and an end tag at the same time. Such a tag is for example <mrow />.)

Every start tag must have a corresponding end tag. Together with the text between them, they form an element. The text between the tags is called the content of the element. The content can consist of other elements or plain text. The name of the element is given by the tags. Consider k=1]]>. This is an element with the name mrow. It contains three elements which have the names mi, mo and mn. We will call an element with the a name A an A element: Our mrow element contains an mi, an mo and an mn element.

We can see now that the elements form a tree structure. Indeed, in the case of k=1]]>, we have an mrow element as the root of the tree. It has 3 children, an mi element, an mn element, and an mn element. The children are ordered and changing their order changes the meaning. The mrow element is called the parent of the mi, mo, and mn elements. The children of an element, their children, as well as all further ones below are called descendants of the element. The parent, grandparent, and any further ones above are called ancestors of the element. In the example above, the ∑]]> is called a descendant of the munderover element it is contained in because it is its child. It is also a descendant of the outermost mrow element. On the other hand, munderover element is an ancestor of the ∑]]> and so is the outermost mrow element. The math element is the root and therefore an ancestor of all other elements and all other elements are its descendants.

But there is more: Every element can have attributes. Attributes are just key-value pairs. The same key can only be used in one of the attributes on the same element. Unlike for children of an element, the order of the attributes does not matter. An attribute has the form key="value" and is placed in the start tag of the element. In the example you can spot an mfrac element wich has a linethickness attribute with value 0.

Identifiers, operators and numbers

In LaTeX, if we want to have a φ, we can write a φ, without marking it in a special way. LaTeX then figures out on its own that it has to set the φ as a variable, not as an operator. In MathML, we have to be explicit: We must put the φ into an mi element, like this: <mi>φ</mi>. For an identifier we use an mi element. For an operator we have to use an mo element. The mn element is for literal numbers. (Literal means written number such as 0, 1, and 3.141. An mn element is not suited for a variable a even if a∈ℝ. In this case, you must use mi.)

The mo element is very powerful. It is not only used for operators like +, · and the like, but it is also used for fences, arrows, separators and more. It is even used for accents like a hat over a variable. The mo element has many attributes which control the behaviour of the operators. However, these attributes have default values depending on the content of the mo element. Every MathML renderer has an operator dictionary which tells it the default values if it encounters an mo element containing a known operator. But it can be that you wish to change these defaults or that you have to set them because you use an operator that is not found in the dictionary. You can look up the operator dictionary in the MathML specification. You can read about the attributes in the section about mo in the MathML specification. It is quite a big section, you may want to read it after you have gone through the rest of this introduction.
